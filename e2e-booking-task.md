# Задача: E2E-тесты базового сценария бронирования

## Цель

Покрыть e2e-тестами основной пользовательский сценарий приложения: гость выбирает тип события, бронирует свободный слот, видит подтверждение успешного бронирования, а владелец календаря видит созданную запись в админке.

Сценарий нужно проверить для двух типов событий:

- встреча на 30 минут;
- встреча на 60 минут.

## Общие предпосылки

Перед выполнением тестов в системе должны быть подготовлены тестовые данные:

- тип события с длительностью `30 мин`;
- тип события с длительностью `60 мин`;
- свободный будущий слот длительностью `30 мин`;
- свободный будущий слот длительностью `60 мин`.

Слоты для сценариев должны быть непересекающимися по времени, так как приложение запрещает создавать бронирование, если его интервал пересекается с любым уже существующим бронированием, даже для другого типа события.

Рекомендуется использовать уникальные имя и email для каждого запуска теста, например с timestamp или случайным суффиксом.

## Сценарий 1: бронирование события на 30 минут

### Тестовые данные

- Тип события: любой доступный тип события с длительностью `30 мин`.
- Имя гостя: `E2E Guest 30`.
- Email гостя: `e2e-30-{unique}@example.com`.
- Слот: свободный будущий слот длительностью `30 мин`.

### Шаги

1. Открыть главную страницу приложения `/`.
2. Нажать ссылку или кнопку `Записаться`.
3. Убедиться, что открылась страница выбора типа события `/book`.
4. Убедиться, что на странице есть заголовок `Выберите тип события`.
5. Выбрать карточку типа события с длительностью `30 мин`.
6. Убедиться, что открылась страница бронирования выбранного события `/book/:eventTypeId`.
7. Убедиться, что на странице отображается выбранный тип события и длительность `30 мин`.
8. Ввести имя гостя в поле `Ваше имя`.
9. Ввести email гостя в поле `Email`.
10. Выбрать дату, на которую есть свободный слот длительностью `30 мин`.
11. Выбрать свободный слот в блоке `Статус слотов`.
12. Нажать кнопку `Продолжить`.

### Ожидаемый результат

После нажатия `Продолжить` пользователь видит уведомление:

```text
Бронирование создано. Мы скоро свяжемся с вами.
```

После перехода в админку `/admin` пользователь видит страницу `Предстоящие встречи`.

В таблице предстоящих встреч есть созданная запись, в которой:

- тип события соответствует выбранному событию на `30 мин`;
- дата и время соответствуют выбранному слоту;
- в колонке `Гость` отображается имя `E2E Guest 30`.

Email в текущем UI админки отдельно не отображается, если у бронирования заполнено имя. Поэтому в e2e-проверке через UI нужно проверять имя гостя, а не одновременное отображение имени и email.

## Общие проверки для обоих сценариев

В каждом сценарии тест должен подтвердить, что:

- пользователь может пройти путь от главной страницы до страницы бронирования;
- список типов событий загружается и содержит нужную длительность;
- выбранный тип события открывает страницу бронирования;
- выбранный слот доступен до создания бронирования;
- кнопка `Продолжить` становится доступной после выбора слота;
- после успешного бронирования появляется уведомление об успехе;
- созданное бронирование отображается в админке;
- в админке отображается имя гостя, указанное при бронировании.

## Что не входит в базовый сценарий

Эта задача не покрывает:

- конфликты бронирований;
- ошибки валидации;
- отсутствие доступных слотов;
- ошибки API;
- проверку email в админке через UI;
- авторизацию, так как в текущем MVP ее нет.
